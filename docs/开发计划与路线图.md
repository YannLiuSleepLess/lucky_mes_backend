## 1. 项目现状评估

### 1.1 已实现模块

| 模块 | 完成度 | 说明 |

| **产品管理** | 90% | Product聚合根、规格值对象、光伏特有字段(IsSerialTraced/YieldRate) |

| **工程变更(ECN)** | 85% | 状态机(Draft→PendingReview→Approved→Executed)、紧急变更、分布式事件(待启用) |

| **领域服务** | 80% | 产品版本对比、BOM完整性校验、循环引用检测、生效版本获取 |
| **数据种子** | 100% | 完整光伏产业链数据（8原材料→3半成品→2成品）+ 工艺路线 |

| **API暴露** | 100% | ABP自动生成API：Product、EngineeringChange、ProcessRoute的完整CRUD |

### 1.2 当前存在的问题

| 问题 | 严重程度 | 说明 |
|------|----------|------|
| 测试严重不足 | 高 | 所有测试项目仅含示例代码，无业务逻辑测试 |
| 权限系统未配置 | 高 | `AbpDemoPermissions` 和 `PermissionDefinitionProvider` 为空 |
| 分布式事件未启用 | 中 | ECN事件处理器仅输出Console日志，RabbitMQ未配置 |
| 泛型仓储未充分利用 | 低 | ProductAppService使用`IRepository<Product>`而非自定义`IProductRepository` |
| BOM项更新方式笨拙 | 低 | 采用"先删除再添加"方式，建议改为直接修改 |
| 缺失核心MES业务 | 极高 | 工单、质量、设备、洁净车间等核心功能均未实现 |

## 3. 技术规范与约定

### 3.1 代码规范

```
命名规范:
- 聚合根: 名词，如 Product, WorkOrder
- 领域服务: 动词+名词，如 BomConversionService
- 应用服务: 名词+AppService，如 ProductAppService
- 仓储接口: I+名词+Repository，如 IProductRepository
- 领域事件: 名词+Event，如 WorkOrderReleasedEvent
- DTO: XxxDto / CreateXxxRequest / UpdateXxxRequest

文件组织:
- 按业务领域分组，不按技术类型分组
- 每个领域一个文件夹，含Aggregates/Entities/Events/ValueObjects子目录
- 领域事件放在 Events/ 子目录
- DTO放在 Dtos/ 子目录
```

### 3.2 数据库规范

```sql
-- 表命名：复数形式，蛇形命名，加前缀
AbpDemoProducts
AbpDemoWorkOrders
AbpDemoWorkOrderRoutings
AbpDemoQualityInspections

-- 唯一索引
CREATE UNIQUE INDEX IX_Products_ProductCode ON AbpDemoProducts(ProductCode);
CREATE UNIQUE INDEX IX_WorkOrders_OrderNo ON AbpDemoWorkOrders(OrderNo);

-- 常用查询索引
CREATE INDEX IX_WorkOrders_Status ON AbpDemoWorkOrders(Status);
CREATE INDEX IX_WorkOrders_ProductId ON AbpDemoWorkOrders(ProductId);
CREATE INDEX IX_WorkOrderRoutings_WorkOrderId ON AbpDemoWorkOrderRoutings(WorkOrderId);
CREATE INDEX IX_EquipmentDataLogs_EquipmentId_LogTime ON AbpDemoEquipmentDataLogs(EquipmentId, LogTime);
```

### 3.3 领域设计原则

1. **聚合根边界**：
   - Product (聚合根) → ProductVersion (子实体) → BomItem (子实体的子实体)
   - WorkOrder (聚合根) → WorkOrderRouting (子实体) + WorkReport (子实体)
   - QualityInspection (聚合根) → InspectionItem (子实体)

2. **跨聚合引用**：只能通过ID引用，不能直接引用内部实体

3. **领域事件**：聚合根内部通过 `AddDomainEvent()` 发布，应用层或基础设施层处理

4. **业务规则位置**：
   - 聚合根内部不变量 → 聚合根方法
   - 跨聚合协调 → 领域服务
   - 输入校验 → 应用层Validator

### 3.4 光伏特有注意事项

| 特性 | 实现位置 | 注意点 |
|------|----------|--------|
| **单片追溯** | WorkReport.SerialNos(JSON) + ProductSerialNo/Mapping表 | Phase 1用JSON快速上线，Phase 4迁移到独立表 |
| **一对多转换** | BomItem.YieldRate + WorkOrderRouting.Input/OutputQuantity | 切片工序注意单位换算 |
| **分选工序** | ProcessStep.IsSortingStep + ThresholdA/B/C + NextStepForGradeA/B/C | 等级判定在应用层服务实现 |
| **炉次管理** | WorkReport.BatchNo | 扩散/PECVD等批量工序使用 |
| **SPC控制限** | ProcessParameter.UCL/LCL/USL/LSL | 超差时触发QualityAlertEvent |

## 4. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| **WorkOrder状态机复杂度高** | 工期延误 | 先实现基础状态流转，复杂场景（部分报工、分批完工）放到Phase 2后 |
| **序列号追溯数据量大** | 性能问题 | Phase 1先用JSON字段，Phase 4再考虑独立表+分区策略 |
| **设备数据日志高频写入** | 数据库压力 | 预留EquipmentDataLog独立存储方案，考虑时序数据库 |
| **测试编写耗时** | 质量风险 | 每个Phase结束时必须完成单元测试，不追求100%但核心逻辑必须覆盖 |
| **ABP版本升级** | 兼容性问题 | 锁定8.3.4版本，Phase 5再评估升级 |
